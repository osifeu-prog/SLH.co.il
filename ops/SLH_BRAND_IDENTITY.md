## DESIGN PRINCIPLES

### 1. Clarity
- Every element has a purpose
- No decoration without function
- Clear information hierarchy
- Obvious interaction points
- Never hide important information

**Implementation:**
- Use size, color, position to emphasize
- Remove visual noise
- Test with users unfamiliar with SLH
- Explain complex concepts simply

---

### 2. Authenticity
- Real, not fake
- Show the machinery
- Honest about limitations
- Transparent about costs
- Visible technical depth

**Implementation:**
- Use ASCII art and monospace fonts
- Show transaction details
- Explain fee structures
- Don't hide complexity, explain it
- Show real-time updates

---

### 3. Accessibility
- Works for everyone
- Multiple ways to navigate
- Color-blind safe palettes
- Keyboard shortcuts available
- Respects user preferences

**Implementation:**
- WCAG 2.1 AA compliance minimum
- Test with screen readers
- Provide alt text
- Don't rely on color alone
- Support high contrast mode

---

### 4. Consistency
- Same patterns everywhere
- Familiar interactions
- Unified visual language
- Same voice and tone
- Predictable behavior

**Implementation:**
- Use component library
- Document all patterns
- Train team on standards
- Review for consistency
- Refactor inconsistencies

---

### 5. Respect
- User time is valuable
- No unnecessary steps
- No dark patterns
- Fast responses
- Clear expectations

**Implementation:**
- Batch operations
- Pre-fill information
- Minimize clicks
- Show processing state
- Deliver quickly

## BRAND VOICE & TONE

### Core Characteristics

**Direct**
- No BS, no fluff
- Say what you mean
- Be honest about tradeoffs
- Admit when you don't know
- Answer the real question

**Technical but Accessible**
- Use precise language
- Explain jargon
- Show the mechanics
- Don't talk down to users
- Assume intelligence

**Confident**
- Stand behind decisions
- Speak with authority
- But stay humble
- Show your work
- Invite questions

**Fair**
- Apply rules equally
- Explain penalties
- Offer paths forward
- Listen to feedback
- Change when needed

**Mysterious**
- Don't explain everything
- Leave some wonder
- Use metaphors (protocol, forge, vault)
- Hint at deeper complexity
- Make discovery rewarding

---

### Language Guidelines

**Good Examples:**
✅ "Earn rewards by staking your tokens in our vaults"
✅ "Market detected unusual activity. Your account is temporarily frozen. Appeal here."
✅ "You've been marked with 10 ZUZ for attempting fraud. Complete redemption course to reduce this."
✅ "Transfer successful. Transaction ID: 0x7a9..."

**Bad Examples:**
❌ "Utilize our decentralized smart contract platform for optimal yield optimization"
❌ "Suspicious activity detected. Access denied." (no explanation)
❌ "System error." (too vague)
❌ "Your action was processed." (passive voice)

---

### Tone Variations

**Success/Positive:**
- Affirming, celebratory
- Use active voice
- Acknowledge the achievement
- Example: "You've earned 100 ZVK! Stake them to multiply your gains."

**Error/Negative:**
- Direct, explanatory
- Tell user what happened
- Explain how to fix
- Example: "Transfer failed: Insufficient balance. You need 50 MNH more."

**Warning/Caution:**
- Urgent but measured
- Explain the risk
- Provide options
- Example: "Detected unusual activity. Is this transfer to a new address? Confirm or cancel."

## MOTION PRINCIPLES

### Animation Guidelines

**Duration:**
- Quick interactions: 0.3s
- Medium interactions: 0.6s
- Long animations: 1-2s
- Never more than 2 seconds (feels slow)

**Easing:**
- Default: ease-out (feels natural)
- Entrance: ease-out (accelerates in)
- Exit: ease-in (slows down)
- Avoid linear (feels mechanical)

**Performance:**
- 60fps minimum (smooth)
- Respect prefers-reduced-motion
- Use transform/opacity (GPU-accelerated)
- Avoid layout thrashing

**Purpose:**
- Provide feedback (something changed)
- Guide attention (show where to look)
- Provide context (connection between states)
- Never decorative alone

---

### Standard Transitions

**Page to Page:**
```
Fade + Slide-right
Duration: 0.6s
Easing: ease-out
Purpose: Clear transition, shows progression
```

**Element Appearance:**
```
Fade-in
Duration: 0.4s
Easing: ease-out
Purpose: Gentle introduction
```

**Hover State:**
```
Scale 1.05 + Glow
Duration: 0.2s
Easing: ease-out
Purpose: Shows interactivity
```

**Click/Press:**
```
Scale 0.95 + Flash
Duration: 0.1s + 0.2s
Easing: ease-out
Purpose: Haptic-like feedback
```

**Error:**
```
Shake 3x + Red pulse
Duration: 0.4s + 0.6s
Easing: ease-out
Purpose: Attention, urgency
```

**Success:**
```
Bounce-in + Green glow
Duration: 0.5s
Easing: ease-out
Purpose: Celebration
```

## DO'S & DON'TS

### Visual Design

#### DO:
✅ Use neon colors (cyan, magenta, purple)
✅ Use monospace fonts (Courier, Roboto Mono)
✅ Use ASCII art and grid patterns
✅ Be technical and show complexity
✅ Use dark backgrounds (#0A0A1A)
✅ Make interactions snappy (0.3-0.6s)
✅ Show real-time data updates
✅ Use 3D effects sparingly
✅ Respect user preferences (dark mode, etc)
✅ Test for accessibility (WCAG AA minimum)

#### DON'T:
❌ Use gradients (too soft)
❌ Use serif fonts (too formal)
❌ Use stock photos (inauthentic)
❌ Hide complexity (explain it)
❌ Make animations too long
❌ Use pure black (#000000)
❌ Use light backgrounds
❌ Make buttons too small
❌ Overuse animation (decoration)
❌ Ignore accessibility

---

### Language & Tone

#### DO:
✅ Be direct and honest
✅ Explain technical concepts
✅ Use active voice
✅ Show system state
✅ Explain errors clearly
✅ Give users options
✅ Use metaphors (forge, vault, etc)
✅ Show respect for users
✅ Be consistent
✅ Admit uncertainty

#### DON'T:
❌ Use jargon without explanation
❌ Be vague or evasive
❌ Use marketing speak
❌ Hide what's happening
❌ Give cryptic error messages
❌ Assume user knowledge
❌ Use passive voice
❌ Be sarcastic or flippant
❌ Vary tone unexpectedly
❌ Pretend to know everything

---

### Interaction

#### DO:
✅ Provide clear feedback
✅ Use confirmation for important actions
✅ Enable undo when possible
✅ Show processing state
✅ Respect keyboard users
✅ Support common shortcuts
✅ Save user progress
✅ Show loading indicators
✅ Batch operations
✅ Explain restrictions

#### DON'T:
❌ Perform risky actions silently
❌ Require confirmation for everything
❌ Make undo impossible
❌ Leave users guessing
❌ Hide keyboard shortcuts
❌ Use obscure shortcut keys
❌ Lose unsaved work
❌ Show no progress indication
❌ Require one action at a time
❌ Restrict without explanation
